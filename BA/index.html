    📌 Business Analyst Requirements Gathering Roadmap

    1. Understand the Business Context

    • Meet with stakeholders to understand goals, challenges, and project purpose.
    • Review documents: business cases, current process flows, previous audits.
    • Tools: SWOT Analysis, PESTLE, Porter’s 5 Forces.

    2. Identify Stakeholders

    • Identify everyone impacted: users, developers, compliance, operations, etc.
    • Create a stakeholder map or RACI matrix.
    • Document roles, responsibilities, and communication preferences.

    3. Elicit Requirements

    Use a variety of techniques: - Workshops – Interactive sessions with teams. - Interviews – One-on-one sessions with SMEs and end-users. - Surveys – Collect insights from a large audience. - Observation – Shadow users to understand tasks and challenges. - Document Analysis – Review legacy systems, logs, and reports. - Prototyping – Low-fidelity mockups to get early feedback.

    4. Analyze and Validate Requirements

    • Categorize requirements: functional, non-functional, technical, business rules.
    • Use models:
    • Use Case Diagrams
    • User Stories
    • Process Flows
    • UML
    • Validate with stakeholders to ensure accuracy and alignment.

    5. Document Requirements

    Document in formats such as: - Business Requirements Document (BRD) - Functional Requirements Specification (FRS) - User Stories / Epics (for Agile) - Requirements Traceability Matrix (RTM)

    Each should include: - Requirement description - Acceptance criteria - Priority and dependencies

    6. Get Sign-off

    • Present documentation for review and feedback.
    • Obtain formal approval from stakeholders and project sponsors.

    7. Support Development and Testing

    • Clarify requirements during sprints or development phases.
    • Participate in:
    • Daily Stand-ups
    • Sprint Planning
    • Backlog Grooming
    • Help QA teams align test cases with requirements.
    • Participate in UAT and validation sessions.

    8. Manage Changes

    • Log and review any change requests.
    • Update documentation and traceability.
    • Communicate changes to affected stakeholders.

    📄 Business Requirements Document (BRD)

    1. Project Overview

    • Project Name: [Enter project name]
    • Sponsor: [Enter sponsor name]
    • Prepared by: [BA Name]
    • Date: [MM/DD/YYYY]

    2. Business Objectives

    • Objective 1: [Brief description of what the project aims to achieve]
    • Objective 2: [...]

    3. Background and Context

    • Description of current process or situation
    • Why the project is needed

    4. Scope

    • In-Scope:
    • [Item 1]
    • [Item 2]
    • Out-of-Scope:
    • [Item 1]
    • [Item 2]

    5. Stakeholders

    Name Role Contact
    John Doe Product Owner john.doe@example.com
    Jane Smith QA Lead jane.smith@example.com

    6. Business Requirements

    ID Requirement Priority Source
    BR-01 Users must log in using SSO. High Security Team
    BR-02 The system should generate weekly reports Medium Sales Team

    7. Functional Requirements

    ID Function Description
    FR-01 Login Module Users authenticate via company SSO.
    FR-02 Report Generation Generate report in PDF every Friday.

    8. Non-Functional Requirements

    • Availability: 99.9% uptime
    • Performance: Reports must generate in under 2 minutes
    • Compliance: Must meet GDPR requirements

    9. Assumptions & Constraints

    • Data is available in the current system
    • Project team resources will be available full-time

    10. Approval

    Name Title Signature Date

    🧩 Use Case: Login to System

    Actors:
    - User
    - Authentication Server

    Basic Flow: 1. User opens the application.
    2. User selects "Login with SSO."
    3. System redirects to Identity Provider.
    4. User enters credentials.
    5. Identity Provider validates and returns token.
    6. System grants access.

    Alternate Flow:
    - If credentials are invalid, display error message.

    Preconditions:
    - User has valid credentials.

    Postconditions:
    - User is authenticated and redirected to dashboard.
